# UX истекшей авторизации

Когда пользователь больше не авторизован, интерфейс не должен показывать технические причины вроде `401`, `419`, `CSRF`, `token expired` или `session expired`.

## Пользовательское поведение

- На публичной странице состояние аккаунта очищается тихо.
- В личном кабинете пользователь мягко возвращается на страницу входа.
- Текст для пользователя: `Пожалуйста, войдите в аккаунт, чтобы продолжить.`
- После успешного входа пользователь возвращается на страницу, с которой его отправили на авторизацию.

## Зачем это нужно

Пользователю не важно, закончилась ли cookie-сессия, сломался CSRF-token или backend вернул unauthorized status. Важно понятное действие: снова войти и продолжить работу без ощущения системной ошибки.

## Технический контракт

- `401` и `419` обрабатываются единым frontend-событием авторизации.
- Локальное состояние пользователя очищается сразу.
- Для `/account/*` выполняется redirect на `/sign-in?redirectTo=...`.
- Toast или message не должен содержать технических терминов.
